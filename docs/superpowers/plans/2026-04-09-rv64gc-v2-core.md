# RV64GC v2 Core Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement a 6-wide out-of-order RV64GC core that achieves 2.5+ IPC on CoreMark, from clean-sheet RTL guided by gem5 sweep data.

**Architecture:** Bottom-up build: packages, functional units, backend structures, rename/dispatch, frontend, memory, integration. Each phase produces independently testable modules. Integer pipeline first; FP deferred until integer achieves IPC targets on CoreMark.

**Tech Stack:** SystemVerilog (IEEE 1800-2017), Verilator 5.x for simulation, GCC 15.x riscv64-unknown-elf cross-compiler, GNU Make.

**v1 reference:** `D:/agent-workspace/RV64GC/` (may be renamed to `rv64gc-v1`). Use for naming conventions and interface patterns. Do NOT carry over architectural bottlenecks (group-hold rename, 2-wide dispatch, single-select IQ, arch-RAT recovery).

**Spec:** `doc/rv64gc_v2_uarch.md` is the authoritative microarchitecture reference. Read it before starting any task.

    localparam int ARCH_REGS = 32;
    localparam int ARCH_REG_BITS = 5;

    localparam int INT_PRF_DEPTH  = 256;
    localparam int FP_PRF_DEPTH   = 128;  // v1: 80
    localparam int PHYS_REG_BITS  = $clog2(INT_PRF_DEPTH);  // 8 bits

    localparam int ROB_DEPTH      = 192;

    localparam int INT_FREE_LIST_DEPTH = INT_PRF_DEPTH - ARCH_REGS;  // 224

    localparam int NUM_INT_IQS    = 3;

    localparam int IQ_MEM_DEPTH   = 32;

- [ ] **Step 1: Write ALU testbench**

The testbench must verify:
- All base RV64I ALU ops (ADD, SUB, AND, OR, XOR, SLT, SLTU, SLL, SRL, SRA, LUI)
- All W-suffix variants (ADDW, SUBW, SLLW, SRLW, SRAW) with sign extension
- Zba: sh1add, sh2add, sh3add (plus .uw variants)
- Zbb: min/max, clz/ctz/cpop, andn/orn/xnor, rol/ror, sext.b/h, rev8, orc.b
- Zbs: bset/bclr/binv/bext
- Zicond: czero.eqz, czero.nez
- Edge cases: x=0, x=MAX, x=MIN, shift by 0/63

Testbench structure: drive `operand_a`, `operand_b`, `op`, `is_w_op`, check `result` against expected value.

- [ ] **Step 2: Implement ALU**

Port list (same structure as v1, wider `op` field):
```systemverilog
module alu
    import rv64gc_pkg::*;
    import uarch_pkg::*;
(
    input  logic [63:0] operand_a,
    input  logic [63:0] operand_b,
    input  alu_op_e     op,
    input  logic        is_w_op,
    output logic [63:0] result
);
```

Key implementation details per extension:
- **Zba sh*add:** `(rs1 << N) + rs2` -- just a shifted add, one case per N
- **Zba .uw variants:** zero-extend rs1[31:0] before shift-add
- **Zbb CLZ/CTZ:** priority encoder. CLZ: `casez` on leading 1 bit position. CTZ: reverse bits then CLZ. Handle 64-bit and 32-bit (W) variants.
- **Zbb CPOP:** popcount via sum of bit slices
- **Zbb min/max:** comparator + mux. Signed and unsigned variants.
- **Zbb ROL/ROR:** `(a << shamt) | (a >> (64-shamt))`. Handle W variants.
- **Zbb REV8:** swap byte order: `{a[7:0], a[15:8], ..., a[63:56]}`
- **Zbb ORC.B:** OR-combine each byte: `{8{|a[63:56]}, 8{|a[55:48]}, ...}`
- **Zbs:** `1 << shamt` to create bit mask, then AND/OR/XOR/extract
- **Zicond:** `(rs2 == 0) ? 0 : rs1` and `(rs2 != 0) ? 0 : rs1`

- [ ] **Step 2: Implement BRU**

```systemverilog
module bru
    import rv64gc_pkg::*;
    import uarch_pkg::*;
(
    input  logic [63:0] operand_a,     // rs1 value
    input  logic [63:0] operand_b,     // rs2 value
    input  logic [63:0] pc,
    input  logic [63:0] imm,
    input  br_op_e      op,
    input  logic        is_fused,      // fused compare-and-branch

    output logic [63:0] result,        // rd value (PC+4 for JAL/JALR)
    output logic        taken,
    output logic [63:0] target,
    output logic        mispredict
);
```

Fused compare-and-branch: when `is_fused=1`, the BRU performs SLT/SLTU comparison and branch resolution in one cycle. `fusion_type` selects: `{slt, sltu} x {beq_x0, bne_x0}`.

    input  logic        is_w_op,

- [ ] **Step 2: Implement rename**

```systemverilog
module rename
    import rv64gc_pkg::*;
    import uarch_pkg::*;
(
    input  logic clk, rst_n,
    input  decoded_insn_t dec_insn [0:PIPE_WIDTH-1],
    input  logic [2:0]    dec_count,
    output renamed_insn_t ren_insn [0:PIPE_WIDTH-1],
    output logic [2:0]    ren_count,
    input  logic [ROB_IDX_BITS-1:0] rob_alloc_idx [0:PIPE_WIDTH-1],
    input  logic                    rob_alloc_ready,
    output logic stall,
    input  logic dq_full,
    input  logic [INT_PRF_DEPTH-1:0] preg_ready_table,
    input  flush_t flush_in,
    input  commit_t commit_in [0:PIPE_WIDTH-1],
    input  logic [2:0] commit_count
);
```

**Per-slot independent advance:**
```
for each slot i (0 to 5):
    if slot i valid AND free_list has entry AND ROB has entry AND DQ has space:
        rename slot i (read RAT for rs1/rs2, allocate PRF for rd, write RAT)
        advance (clear valid, pass to dispatch)
    else:
        hold slot i (retain for next cycle)
```

**Move elimination** (checked before PRF allocation):
- `mv rd, rs` -> copy RAT mapping, mark ROB complete at rename, no PRF alloc
- `li rd, 0` / `xor rd, rd, rd` -> map rd to physical zero register, mark complete

## Implementation Notes

### Module Testing Strategy

Each module gets a Verilator testbench with:
1. **tb_\<module\>.sv** -- SV testbench: instantiates DUT, drives inputs, checks outputs
2. **tb_\<module\>.cpp** -- C++ driver: clock toggle, reset, cycle limit, VCD trace

Reusable C++ driver pattern:
```cpp
#include <verilated.h>
#include <verilated_vcd_c.h>
#include "Vtb_<module>.h"
int main(int argc, char** argv) {
    Verilated::commandArgs(argc, argv);
    Verilated::traceEverOn(true);
    auto* top = new Vtb_<module>;
    auto* tfp = new VerilatedVcdC;
    top->trace(tfp, 99);
    tfp->open("sim.vcd");
    top->clk = 0; top->rst_n = 0;
    for (int i = 0; i < 20; i++) { top->clk = !top->clk; top->eval(); tfp->dump(i); }
    top->rst_n = 1;
    int cycle = 0;
    while (cycle < 10000 && !Verilated::gotFinish()) {
        top->clk = 1; top->eval(); tfp->dump(20 + cycle*2);
        top->clk = 0; top->eval(); tfp->dump(20 + cycle*2 + 1);
        cycle++;
    }
    tfp->close(); delete tfp; delete top;
    return 0;
}
```

### Build Verification at Each Phase

After each phase: lint-only all new modules, run module testbenches, no regressions.

### IPC Debugging

If CoreMark IPC is below target, add performance counters and check:
- `rename_stall_cycles` -- should be <10% (was 68% in v1)
- `hold_valid_cycles` -- should be <2% (was 57% in v1)
- `rob_head_blocked_cycles` -- should be <10%
- `fetch_stall_cycles` -- should be <2%
- `branch_mpki` -- should be <0.5 with TAGE-SC-L
- `fusion_rate` -- dynamic fusion percentage

### FP Pipeline (Deferred)

Integer pipeline is the priority. FP (RV64FDC) deferred because CoreMark and Dhrystone do not use FP. FP PRF, FP IQ, FPU, and FP decode paths are independent and can be added after integer IPC targets are met.
