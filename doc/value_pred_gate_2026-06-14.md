# Value-Prediction Feasibility Gate — chain band (2026-06-14)

**Verdict: KILL.** Value prediction is **DEAD for the rv64gc-v2 chain band.** Every
binding loop-carried recurrence in the chain band is, on a *real* (cold / live-input)
basis, **0–2% confidently-predictable** by *any* practical value predictor
(last-value, stride, 2-delta, FCM-order-1/2, VTAGE-style context). The chain band is
hereby **certified as truly irreducible on a scalar core** — it is closed even to the
one lever (value speculation) the structural studies left as theoretically open.

This is OFFLINE-ONLY: value streams generated analytically from the disassembled
binaries (the binding recurrences are deterministic functions of the binary + known
init state), replayed through a validated Python predictor suite
(`tools/value_pred_model.py`). **Zero RTL. No sims run.** The Verilator Linux boot
(pid 2196358) was left untouched.

## 2. Offline predictability model (`tools/value_pred_model.py`)

Predictor suite (industry-standard VP designs): **last-value** (Gabbay/Mendelson),
**stride** + **2-delta-stride** (Eickemeyer), **FCM-order-1/2** (Sazeides/Smith
finite-context), **VTAGE-style** value-history context (Perais/Seznec). Each entry
carries a saturating confidence counter; a prediction is "confident" (issued
speculatively, the only case that can collapse the chain) only at counter ≥ thresh;
a wrong confident prediction resets confidence (models the selective-replay cost).

**Model validated** against known streams: constant-stride → stride 99% (last-value /
FCM correctly 0%); constant → last-value 99%; 3-cycle repeat → FCM 98%. The
zero-results below are real, not implementation artifacts.

### 2a. crc32 LCG seed — the tantalizing target, tested honestly

| Predictor | coverage | accuracy | payoff (=cov×acc) |
|---|---:|---:|---:|
| last-value | 0.0000 | — | **0.0000** |
| stride | 0.0000 | — | **0.0000** |
| 2-delta | 0.0000 | — | **0.0000** |
| FCM-1 / FCM-2 / VTAGE | 0.9529 | 1.0000 | **0.9529** ← ARTIFACT |

The 95% FCM/VTAGE number is **100% a benchmark-reseed artifact, not value
prediction.** Proof, three independent ways:

1. **Cold first pass (first 1024 iters, before any memorization): payoff = 0.0000.**
   The LCG produces 1024 distinct values; FCM has never seen any context, predicts
   nothing.
2. **No-reseed (a real long-period PRNG with a live seed): payoff = 0.0000** on *all*
   predictors. Every value over the run is distinct (distinct-fraction 1.0), so no
   FCM context ever recurs.
3. **Finite-table cliff:** the 95% appears only when the value-indexed table holds
   **≥ the full 1024-distinct working set.** Table size 256/512 → **payoff 0.0000**
   (thrash); 1024+ → 0.9529. It is pure memoization of a fixed re-run.

Stride/2-delta are dead by construction: the LCG first-differences
(`1.4e9, −7.5e8, +7.9e8, …`) and second-differences are large and non-constant — an
LCG is *deterministic but not stride-predictable* (the multiplicative recurrence
has no constant delta), exactly as the task hypothesized.

### 2b. aha-mont64 modul64 accumulator — identical signature

| Predictor | cold-pass payoff | steady-state payoff |
|---|---:|---:|
| last-value / stride | 0.0000 | 0.0000 |
| FCM-1/2 / VTAGE | **0.0000** | 0.9830 (ARTIFACT) |

64 distinct accumulator values per reduction; **0% on the cold pass**, 98% only
because the *identical* 64-value sequence is recomputed 472× from fixed inputs. Same
memoization artifact as crc32. (mont64 is in any case **misp-dominated**, 43% of
cycles — the chain is not even its primary binder.)

### 2c. Pointer-chase rows (towers, sglib)

| Regime | FCM-1 | FCM-2 | VTAGE |
|---|---:|---:|---:|
| Realistic data-dependent LIFO `next`-ptr | payoff 0.004 | 0.013 | **0.016** |
| (Unrealistic) strictly-periodic node pool | 0.997 | — | 0.997 |

The carried pointer is reused (small node pool, ~63 distinct addresses) but the
**successor depends on dynamic stack/tree state**, so FCM order-1 aliases (same node
→ different `next`) and confident coverage collapses to **~1–2%**. Only a strictly
periodic pool (which the Hanoi/rbtree mutation pattern is not) would memorize. Stride
is dead (heap addresses, no arithmetic structure).

### 2d. Data-dependent rows (cjpeg, picojpeg, wikisort)

No closed form, but structurally identical to the cold-pass arithmetic case: the
carried values are data-dependent transform/decode/sort state with **no within-run
value recurrence and no stride** (DCT butterflies, Huffman bit-state, merge indices
on LCG-random data). Real predictability ≈ 0; any high FCM number would again be the
fixed-input re-run memoization artifact, not generalizable prediction.

## 4. FUND/KILL verdict

**KILL — value prediction is not a fresh 3.x lever for the chain band.** Concretely:

- **The arithmetic chains (crc32 LCG, mont64 modular) are 0% predictable on a real
  basis.** A long-period LCG and a modular-square recurrence are *deterministic yet
  maximally value-novel*: every carried value is distinct, no stride, no recurring
  context. The task's hypothesis is confirmed — an LCG is deterministic but **not**
  stride/value-predictable.
- **The high FCM/VTAGE numbers are a benchmark memoization artifact** (fixed-input
  re-run + a value table ≥ working-set). They vanish on the cold pass, on the
  no-reseed variant, and at any realistic table size. Building VP RTL to chase them
  would optimize the benchmark's *fixed re-run*, not any real recurrence.
- **The pointer-chase chains (towers, sglib) are ~1–2% predictable** — the carried
  address is reused but the successor is dynamic-state-dependent, so confident
  coverage is negligible.
- **The data-dependent chains (cjpeg, picojpeg, wikisort) carry transform/decode/sort
  state** with no real value recurrence.

This **certifies the chain band as truly irreducible on a scalar OoO core**: it was
already proven a 1-cyc/hop dataflow minimum (`alu.sv` purely combinational); value
prediction was the *only* mechanism that could break the dataflow serialization, and
it is now measured-dead because the carried recurrences are information-theoretically
unpredictable on a real basis. The chain band moves only via **software (chain
shortening / -flto / word-wide idioms)** or **vectorization (a different product on a
different KPI)** — both already in the program. **No RTL is funded.**

**Selective-replay correctness risk (not incurred — recorded for the ledger).** Had
any row been fundable, VP would have required a **new per-value squash mechanism**:
`checkpoint.sv`'s 64 checkpoints are *branch*-triggered, so VP needs verify-at-execute
+ selective replay of the value-mispredicted dependent cone — blast radius into the
FP-operand-bypass path (cf. the 21842ac qNaN bug) and the M1/M2 load paths. That
risk class is **avoided entirely** by this clean offline KILL.
